## 1. บทนำและวัตถุประสงค์

ปัจจุบันไฟล์แนบ (Attachments) ที่เกี่ยวข้องกับงานสนับสนุนด้าน IT ของ CIT มีการจัดเก็บรวมกัน ซึ่งเมื่อเวลาผ่านไป ไฟล์เก่าที่ไม่ได้ใช้งานบ่อยครั้งจะสะสม��พิ่มขึ้น ทำให้เกิดภาระด้านค่าใช้จ่ายในการจัดเก็บข้อมูล (Storage Costs) และอาจส่งผลต่อประสิทธิภาพในการจัดการข้อมูลโดยรวม

ข้อเสนอฉบับนี้มีวัตถุประสงค์เพื่อนำเสนอแนวทางและสถาปัตยกรรมสำหรับระบบจัดเก็บเอกสารเก่า (Archive/Cold Storage System) สำหรับไฟล์แนบที่มีอายุมากกว่า 1 ปี โดยมีเป้าหมายหลักคือ:
*   **ลดค่าใช้จ่าย:** ย้ายไฟล์ที่เข้าถึงน้อยครั้งไปยัง Storage Tier ที่มีราคาถูกกว่า
*   **ปรับปรุงการจัดการข้อมูล:** แยกไฟล์ที่ใช้งานบ่อยออกจากไฟล์ที่เก็บถาวร
*   **รั���ษาความพร้อมใช้งาน:** ยังคงสามารถกู้คืนไฟล์ที่เก็บถาวรได้เมื่อจำเป็น
*   **เพิ่มประสิทธิภาพ:** ลดภาระของระบบจัดเก็บหลัก

## 2. แนวคิด Cold Storage และ File Archival

### 2.1 Cold Storage
**Cold Storage** คือกลยุทธ์การจัดเก็บข้อมูลสำหรับข้อมูลที่มีความถี่ในการเข้าถึงต่ำมาก (Infrequent Access) หรือแทบไม่เข้าถึงเลย แต่ยังคงต้องเก็บรักษาไว้เพื่อวัตถุประสงค์ทางกฎหมาย การตรวจสอบ หรือการอ้างอิงในอนาคต

**คุณสมบัติหลัก:**
*   **ต้นทุนต่ำ:** เป็น Storage Tier ที่มีราคาต่อ GB ถูกที่สุด
*   **ความหน่วงสูง (High Latency):** การเข้าถึงหรือกู้คืนข้อมูลอาจใช้เวลาตั้งแต่ไม่กี่นาทีไปจนถึงหลายชั่วโมงหรือหลายวัน ขึ้นอยู่กับบริการ
*   **ความทนทานสูง:** ออกแบบมาเพื่อความคงทนของข้อมูลในระยะยาว
*   **เหมาะสำหรับ:** ข้อมูลสำรองระยะยาว, ข้อมูลทางประวัติศาสตร์, ข้อมูลเพื่อการปฏิบัติตามข้อกำหนด (Compliance Data)

### 2.2 File Archival Pattern
**File Archival** คือกระบวนการย้ายข้อมูลจากระบบจัดเก็บหลัก (Primary Storage) ไปยังระบบจัดเก็บระยะยาว (Archival Storage) โดยทั่วไปจะใช้สำหรับข้อมูลที่ครบวงจรชีวิตการใช้งานแล้��� แต่ยังจำเป็นต้องเก็บรักษาไว้

**รูปแบบทั่วไป:**
1.  **กำหนดเกณฑ์:** ระบุข้อมูลที่จะเก็บถาวร (เช่น อายุไฟล์ > 1 ปี, สถานะของ Ticket เป็น "Closed" มานาน).
2.  **ย้ายข้อมูล:** ย้ายไฟล์จาก "Hot Storage" (เข้าถึงบ่อย) ไปยัง "Cold Storage" (เข้าถึงน้อย).
3.  **จัดการ Metadata:** เก็บข้อมูลอ้างอิง (เช่น Path ใหม่, วันที่เก็บถาวร) ในฐานข้อมูลหลัก เพื่อให้สามารถค้นหาและกู้คืนได้.
4.  **การกู้คืน:** มีกระบวนการในการกู้คืนไฟล์จาก Cold Storage กลับมาใช้งานเมื่อจำเป็น.

## 3. การวิเคราะห์ Supabase Storage และตัวเลือกการจ���ดเก็บ

Supabase Storage เป็นบริการจัดเก็บไฟล์ที่เข้ากันได้กับ Amazon S3 (S3-compatible) ซึ่งหมายความว่าโดยพื้นฐานแล้ว Supabase Storage Buckets จะทำงานคล้ายกับ S3 Standard Storage Class

### 3.1 ตัวเลือกการบีบอัด/จัดเก็บถาวรใน Supabase
*   **Supabase Storage Buckets:** โดยค่าเริ่มต้น Supabase Storage จะใช้ S3 Standard ซึ่งเป็น Hot Storage ที่มีประสิทธิภาพสูง แต่มีค่าใช้จ่ายต่อ GB สูงกว่า Tier อื่นๆ
*   **การบีบอัด (Compression):** Supabase Storage ไม่มีฟังก์ชันการบีบอัดไฟล์ในตัว การบีบอัดควรทำที่ฝั่ง Client ก่อนอัปโหลดไฟล์ (เช่น การบีบอัดเอกสารเป็น .zip หรือ .gz) เพื่อลดขนา���ไฟล์และค่าใช้จ่ายในการจัดเก็บและถ่ายโอนข้อมูล
*   **การจัดเก็บถาวร (Archiving):**
    *   **แนวทางที่ 1 (Supabase-centric):** สร้าง Supabase Bucket แยกต่างหากสำหรับไฟล์เก่า (เช่น `attachments_archive`) แล้วย้ายไฟล์จาก `attachments_active` ไปยัง `attachments_archive` โดยใช้ Supabase Storage API วิธีนี้ช่วยให้การจัดการยังคงอยู่ใน Supabase Ecosystem แต่ **ไม่ได้ลดค่าใช้จ่ายในการจัดเก็บต่อ GB โดยตรง** เนื่องจากทั้งสอง Bucket ยังคงใช้ S3 Standard (เว้นแต่ Supabase จะมี Tiering Options ในอนาคต) แต่จะช่วยลดค่าใช้จ่ายด้าน Egress และ Request Costs ได้หากไฟล์ใน Archive Bucket ถูกเข้าถึงน้อยลงมาก
    *   **แนวทางที่ 2 (Hybrid - แนะนำ):** ใช้ Supabase Storage (`attachments_active`) สำหรับไฟล์ที่ใช้งานอยู่ และย้ายไฟล์เก่าไปยัง **Amazon S3 Bucket โดยตรง** ที่มีการกำหนดค่า Lifecycle Policies เพื่อใช้ S3 Storage Classes ที่มีราคาถูกกว่า เช่น:
        *   **S3 Intelligent-Tiering:** จัดการ Tiering อัตโนมัติระหว่าง Standard, Standard-IA, Glacier. เหมาะสำหรับข้อมูลที่มีรูปแบบการเข้าถึงไม่แน่นอน.
        *   **S3 Standard-Infrequent Access (S3 Standard-IA):** เหมาะสำหรับข้อมูลที่เข้าถึงน้อยครั้ง แต่ต้องการการกู้คืนที่รวดเร็ว (ภายในมิลลิวินาที).
        *   **S3 Glacier Flexible Retrieval / S3 Glacier Deep Archive:** เหมาะสำหรับข้อ��ูลที่เข้าถึงน้อยมากและยอมรับความล่าช้าในการกู้คืนได้นานขึ้น (หลายนาทีถึงหลายชั่วโมง).
        *   **ข้อดีของแนวทางนี้:** ได้รับประโยชน์สูงสุดจากการลดค่าใช้จ่ายในการจัดเก็บข้อมูลต่อ GB
        *   **ข้อเสีย:** เพิ่มความซับซ้อนในการจัดการ เนื่องจากต้องจัดการ S3 Bucket โดยตรงนอกเหนือจาก Supabase Storage API

**ข้อเสนอแนะ:** สำหรับ CIT IT Support ที่ต้องการลดค่าใช้จ่ายอย่างมีนัยสำคัญ **แนวทางที่ 2 (Hybrid)** โดยใช้ S3 Standard-IA เป็นตัวเลือกที่เหมาะสมที่สุด เนื่องจากให้ความสมดุลระหว่างต���นทุนที่ต่ำลงและการกู้คืนที่รวดเร็วพอสมควร

## 4. สถาปัตยกรรมระบบที่เสนอ

ระบบที่นำเสนอจะเป็นแบบ **อัตโนมัติ (Automatic)** โดยใช้ Cron Trigger เพื่อย้ายไฟล์ตามเกณฑ์ที่กำหนด

### 4.1 สถาปัตยกรรม (Hybrid Approach)

```mermaid
graph TD
    A[Supabase Postgres Database] -- (1) Query files > 1 year --> B(Cron Job: Supabase Edge Function / AWS Lambda)
    B -- (2) Download file from --> C[Supabase Storage: attachments_active (S3 Standard)]
    B -- (3) Upload file to --> D[AWS S3 Bucket: cit-attachments-archive (S3 Standard-IA)]
    B -- (4) Update metadata in --> A
    B -- (5) Delete file from --> C

    E[User/Application] -- (6a) Request active file --> C
    E -- (6b) Request archived file --> A
    A -- (6c) Get archive path --> D
    D -- (6d) Retrieve archived file --> E
```

### 4.2 ส่วนประกอบหลัก

1.  **Supabase Postgres Database:**
    *   เก็บ Metadata ของไฟล์แนบทั้งหมด (เช่น `file_id`, `file_name`, `original_path`, `upload_date`, `is_archived` (boolean), `archive_path` (URL/Key ใน S3 Archive Bucket), `archived_date`).
    *   `upload_date` จะเป็นเกณฑ์หลักในการพิจารณาการเก็บถาวร.
    *   `is_archived` และ `archive_path` จะใช้ในการระบุตำแหน่งของไฟล์.

2.  **Supabase Storage: `attachments_active` Bucket:**
    *   สำหรับจัดเก็บไฟล์แนบที่ใช้งานอยู่ (อายุ < 1 ปี).
    *   ใช้ S3 Standard Storage Class (โดยค่าเริ่มต้นของ Supabase).

3.  **AWS S3 Bucket: `cit-attachments-archive`:**
    *   สำหรับจัดเก็บไฟล์แนบที่เก็บถาวร (อายุ > 1 ปี).
    *   กำหนดค่า Lifecycle Policy ให้ใช้ **S3 Standard-Infrequent Access (S3 Standard-IA)** เพื่อลดค่าใช้จ่าย.
    *   สามารถพิจารณา S3 Intelligent-Tiering ได้หากรูปแบบการเข้าถึงไม่แน่นอน.

4.  **Cron Job (Supabase Edge Function / AWS Lambda):**
    *   เป็น Serverless Function ที่ทำงานตามกำหนดเวลา (เช่น ทุกคืน).
    *   **หน้าที่:**
        *   **ค้นหา:** Query ฐานข้อมูล Supabase Postgres เพื่อหาไฟล์ที่มี `upload_date` เก่ากว่า 1 ปี และ `is_archived` เป็น `false`.
        *   **ดาวน์โหลด:** ใช้ Supabase Storage API เพื่อดาวน์โหลดไฟล์จาก `attachments_active` Bucket.
        *   **อัปโหลด:** ใช้ AWS SDK (เช่น `aws-sdk-js`) เพื่ออัปโหลดไฟล์ไปยัง `cit-attachments-archive` S3 Bucket.
        *   **อัปเดต Metadata:** อัปเดต Record ใน Supabase Postgres โดยตั้งค่า `is_archived` เป็น `true` และบันทึก `archive_path` (S3 Key) และ `archived_date`.
        *   **ลบ:** ใช้ Supabase Storage API เพื่อลบไฟล์ออกจาก `attachments_active` Bucket.
        *   **การจัดการข้อผิดพลาด:** มีกลไกในการจัดการข้อผิดพลาดและบันทึก Log.

### 4.3 กระบวนการกู้คืน (Restoration)

*   **อัตโนมัติ (ผ่าน Application):**
    1.  เมื่อผู้ใช้หรือ Application ต้องการเข้าถึงไฟล์แนบ ระบบจะตรวจสอบ `is_archived` ในฐานข้อมูล.
    2.  ถ้า `is_archived` เป็น `false` ระบบจะดึงไฟล์จาก `attachments_active` Supabase Bucket.
    3.  ถ้า `is_archived` เป็น `true` ระบบจะใช้ `archive_path` เพื่อดึงไฟล์จาก `cit-attachments-archive` S3 Bucket.
    4.  ไฟล์จะถูกส่งกลับไปยังผู้ใช้/Application.
    5.  **ตัวเลือ�� (Rehydration):** สำหรับไฟล์ที่อาจถูกเข้าถึงบ่อยขึ้นชั่วคราวหลังจากการกู้คืน สามารถเพิ่มขั้นตอนการ "Rehydrate" โดยการคัดลอกไฟล์กลับไปยัง `attachments_active` Supabase Bucket ชั่วคราว (และอัปเดต `is_archived` เป็น `false` หรือเพิ่ม `rehydrated_until` timestamp) เพื่อลด Latency ในการเข้าถึงครั้งต่อไป.

*   **กึ่งอัตโนมัติ/Manual (สำหรับแอดมิน):**
    *   แอดมินสามารถเข้าถึง S3 Console โดยตรงเพื่อค้นหาและดาวน์โหลดไฟล์ที่เก็บถาวรได้หากจำเป็น.

## 5. แนวปฏิบัติที่ดีที่สุดสำหรับ Cron Trigger

*   **แพลตฟอร์ม:**
    *   **Supabase Edge Functions:** เหมาะสำหรับงาน���ี่ไม่ซับซ้อนมากนักและต้องการอยู่ใน Supabase Ecosystem. ข้อจำกัดอาจอยู่ที่ Runtime และ Resource.
    *   **AWS Lambda (แนะนำ):** มีความยืดหยุ่นสูง, ปรับขนาดได้อัตโนมัติ, ทนทาน, มีเครื่องมือสำหรับ Monitoring และ Logging ที่ครบครัน (CloudWatch). สามารถตั้งเวลาการทำงานด้วย CloudWatch Events/EventBridge.
    *   **Google Cloud Functions:** คล้ายกับ AWS Lambda, ตั้งเวลาด้วย Cloud Scheduler.
*   **ความทนทาน (Reliability):**
    *   **Idempotency:** การออกแบบ Cron Job ให้สามารถทำงานซ้ำได้โดยไม่ก่อให้เกิดผลลัพธ์ที่ไม่พึงประสงค์ (เช่น การย้ายไฟล์ซ้ำ, การลบไฟล์ที่ไม่มีอยู่).
    *   **Error Handling:** ��ัดการข้อผิดพลาดที่อาจเกิดขึ้นระหว่างการดาวน์โหลด, อัปโหลด, อัปเดตฐานข้อมูล, และลบไฟล์.
    *   **Retry Mechanism:** ใช้กลไกการลองใหม่ (Retry) สำหรับการดำเนินการที่ล้มเหลวชั่วคราว.
*   **การบันทึก (Logging) และการตรวจสอบ (Monitoring):**
    *   บันทึก Log การทำงานทุกครั้ง (สำเร็จ/ล้มเหลว, ไฟล์ที่ถูกย้าย) ไปยัง CloudWatch Logs (สำหรับ Lambda) หรือ Supabase Logs (สำหรับ Edge Function).
    *   ตั้งค่า Alert สำหรับข้อผิดพลาดหรือความล้มเหลวในการทำงาน.
*   **ความปลอดภัย:**
    *   ใช้ IAM Roles (สำหรับ Lambda) หรือ Supabase Service Role Keys (สำหรับ Edge Function) ที่มีสิ���ธิ์ขั้นต่ำที่จำเป็นในการเข้าถึง Supabase Storage, S3 และ Postgres.
    *   ไม่ควร Hardcode API Keys หรือ Credentials.

## 6. กรณีศึกษา CIT: วงจรชีวิตไฟล์และรูปแบบการเข้าถึง

*   **วงจรชีวิตไฟล์:**
    *   **Active Phase (< 1 ปี):** ไฟล์แนบถูกสร้างขึ้นและใช้งานอย่างสม่ำเสมอในช่วง 1 ปีแรก (เช่น การแก้ไขปัญหา, การอ้างอิงเอกสารใน Ticket ที่ยัง Active).
    *   **Archival Phase (> 1 ปี):** หลังจาก 1 ปี การเข้าถึงไฟล์จะลดลงอย่างมาก แต่ยังคงมีความจำเป็นในการเก็บรักษาไว้เพื่อการอ้างอิงในอนาคต, การตรวจสอบ (Audit), หรือกรณีที่ต้องกลับมาดู Ticket เก่า.
*   **รูปแบบการเข้าถึง:**
    *   **Active Files:** ต้องการการเข้าถึงที่รวดเร็ว (Low Latency), ความถี่สูง.
    *   **Archived Files:** ต้องการการเข้าถึงที่ต่ำมาก (Infrequent Access), ยอมรับความล่าช้าในการกู้คืนได้ (เช่น ไม่กี่วินาทีถึงไม่กี่นาที).
*   **ประเภทไฟล์:** เอกสาร (PDF, DOCX), รูปภาพ (JPG, PNG), Log Files, อื่นๆ.

จากรูปแบบนี้ S3 Standard-IA จึงเป็นตัวเลือกที่เหมาะสมที่สุดสำหรับ Cold Storage เนื่องจากมีต้นทุนต่ำกว่า S3 Standard และยังคงให้การกู้คืนที่รวดเร็ว (มิลลิวินาที) ซึ่งเหมาะสำหรับกรณีที่ต้องเข้าถึงไฟล์เก่��อย่างไม่คาดคิด
